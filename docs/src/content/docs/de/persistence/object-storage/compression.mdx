## Konfiguration

```ts
type CompressionAlgo = 'none' | 'gzip' | 'zstd';

interface CompressionConfig {
  algorithm: CompressionAlgo;
  level?:    number;       // algorithmus-spezifisch; geclamped; undefined = Default
}
```

**Level** sind algorithmus-spezifisch.  Out-of-range-Werte werden
geclamped; ohne `level` gilt der Implementierungs-Default:

- **gzip**: 0 (keine Kompression) βÄ™ 9 (am besten).  Default 6.
- **zstd**: 1 (am schnellsten) βÄ™ 22 (am besten).  Default 3.
- **none**: ignoriert.

FΟΦr die meisten Workloads sind die **Defaults in Ordnung** βÄî dreh
nur hoch, wenn Storage-Kosten dominieren.

## Runtime-UnterstΟΦtzung

- **Compress (Write)** βÄî nur nativ: **Bun** (`Bun.zstdCompressSync`)

  nicht kryptisch beim ersten Persist.

## Level wechseln

Das Level ist eine reine **Encoder**-Einstellung.  Es wird **nicht**
auf dem Wire gespeichert (das Manifest hΟΛlt den *Algorithmus* fest,
nicht das Level), und der Decompressor braucht es nie βÄî gzip- und
zstd-Frames sind selbst-beschreibend.

Daher braucht ein **Level-Wechsel keine Migration**: Bodies, die mit
dem alten Level geschrieben wurden, dekodieren weiter, neue Bodies
nutzen das neue Level, und beide mischen sich problemlos.  Du kannst
das Level auch jederzeit per Actor anheben oder senken, ohne
bestehende Daten anzufassen.

<aside>
**zstd Ultra-Level (βâΞ 20) + der `fzstd`-Fallback.**  Ultra-Level
nutzen groΟüe Match-Fenster.  Der Pure-JS-`fzstd`-Decompress-Fallback
deckelt die Backreference-Distanz bei 2¬≤β¹Β (32 MB) und kann

deiner Flotte ohne natives zstd laufen kΟΕnnte (also nicht Bun und
nicht Node βâΞ 22.15), halte das Level **βâΛ 19**.  Auf nativen Lesern

## CPU-Kosten (grobe Richtwerte)

Single-Thread, ~100 KB Blob βÄî nur GrΟΕΟüenordnung; echte Zahlen hΟΛngen
von Payload und Runtime ab:

- **gzip Level 6** βÄî ~1-3 ms Encode, ~0,5 ms Decode.
- **gzip Level 1** βÄî ~0,5 ms Encode, schnelles Decode.

  VerhΟΛltnis**, sehr schnelles Decode.

zstds Asymmetrie (gΟΦnstiges Decode auf jedem Level) macht es zu
einem starken Default fΟΦr lese-lastigen State.  FΟΦr typische
State-Store-Workloads (kleine Objekte, moderate Schreibrate) sind
