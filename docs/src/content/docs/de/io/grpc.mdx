Das Framework stellt zwei Actor-Klassen für gRPC bereit:

| Klasse | Rolle |
| --- | --- |
| **`GrpcServerActor`** | Hostet einen gRPC-Server; stellt Service-Methoden bereit. |
| **`GrpcClientActor`** | Verbindet sich mit einem Remote-gRPC-Service; ruft Methoden auf. |

Beide umschließen `@grpc/grpc-js`.  Nützlich, wenn du
**Protobuf-typisierte Verträge** hast und den Broker-Actor-
Lifecycle (Reconnect, Buffer, Subscriber-Fan-out) für clientseitige
Aufrufe willst.

Beide Actors werden über einen **fluenten Options-Builder**
konfiguriert — `GrpcServerOptions.create()…` /
`GrpcClientOptions.create()…` — und wie jeder andere Actor
gespawnt.

## Server

Der Server ist ein konkreter Actor: spawne ihn direkt.  Er lädt das
Proto, registriert die Methoden-Handler und **bindet in
`preStart`** — der `spawn`-Aufruf kehrt zurück, bevor der Bind
abgeschlossen ist, also gib ihm vor dem ersten Client-Call einen
Moment.

Jede Methode wird auf einen **Handler-Actor** abgebildet, der den
eingehenden Call als Nachricht empfängt.  Handler sind in der
`handlers`-Map nach Methodennamen indiziert; jede Methode, die
nicht in der Map steht, wird als `UNIMPLEMENTED` gemeldet.

```ts
import {
  Actor,
  Props,
  GrpcServerActor,
  GrpcServerOptions,
  type GrpcUnaryCall,
  type GrpcServerStreamCall,
} from 'actor-ts';

// A unary handler: reply once via `respond`.
class GetSensorHandler extends Actor<GrpcUnaryCall> {
  override onReceive(call: GrpcUnaryCall): void {
    const id = (call.request as { id: string }).id;
    call.respond({ id, label: `sensor-${id}` });
  }
}

// A server-stream handler: emit chunks via `send`, finish via `complete`.
class WatchSensorHandler extends Actor<GrpcServerStreamCall> {
  override onReceive(call: GrpcServerStreamCall): void {
    const limit = (call.request as { limit?: number }).limit ?? 5;
    for (let i = 0; i < limit; i++) {
      call.send({ value: 20 + i, ts: Date.now() });
    }
    call.complete();
  }
}

const getHandler = system.spawn(Props.create(() => new GetSensorHandler()), 'get');
const watchHandler = system.spawn(Props.create(() => new WatchSensorHandler()), 'watch');

const grpcServerOptions = GrpcServerOptions.create()
  .withProtoPath(protoPath)
  .withPackageName('sensor.v1')
  .withServiceName('SensorService')
  .withBind('127.0.0.1:50051')
  .withHandlers({
    GetSensor: { kind: 'unary', target: getHandler },
    WatchSensor: { kind: 'serverStream', target: watchHandler },
  });
const server = system.spawn(
  Props.create(() => new GrpcServerActor(
    grpcServerOptions,
  )),
  'grpc-server',
);
```

Die Bind-Adresse ist ein einzelner `host:port`-String über
`withBind`; `packageName` und `serviceName` werden separat gesetzt.
Jeder Handler-Deskriptor ist `{ kind, target }`, wobei `kind`
entweder `'unary'`, `'serverStream'` oder `'bidi'` ist.

### Call-Formen der Handler

Das Framework deserialisiert das Protobuf-Payload und übergibt dem
Handler ein typisiertes Call-Objekt:

| Handler-`kind` | Call-Typ | Reply-API |
| --- | --- | --- |
| `unary` | `GrpcUnaryCall` | `respond(res)` / `respondError(message, code?)` |
| `serverStream` | `GrpcServerStreamCall` | `send(chunk)` / `complete()` / `fail(message, code?)` |
| `bidi` | `GrpcBidiCall` | `onData(target)` / `send(chunk)` / `complete()` |

Jeder Call trägt `method`, den deserialisierten `request` (außer
der Bidi-Call, der seine Requests streamt) und ein
`metadata`-Record.  `respondError` / `fail` nehmen einen optionalen
numerischen Status-Code, der auf `13` (INTERNAL) fällt.

## Client

```ts
import {
  Actor,
  Props,
  GrpcClientActor,
  GrpcClientOptions,
  type GrpcInbound,
} from 'actor-ts';

const grpcClientOptions = GrpcClientOptions.create()
  .withProtoPath(protoPath)
  .withPackageName('sensor.v1')
  .withServiceName('SensorService')
  .withEndpoint('127.0.0.1:50051');
const client = system.spawn(
  Props.create(() => new GrpcClientActor(
    grpcClientOptions,
  )),
  'sensor-client',
);

// A collector actor receives every reply / stream frame as `GrpcInbound`.
class ReplyCollector extends Actor<GrpcInbound> {
  override onReceive(message: GrpcInbound): void {
    if (message.kind === 'reply') {
      console.log('unary reply:', message.response);
    } else if (message.kind === 'stream-data') {
      console.log('stream chunk:', message.chunk);
    } else if (message.kind === 'stream-end') {
      console.log('stream complete');
    } else if (message.kind === 'rpc-error' || message.kind === 'stream-error') {
      console.error('error:', message.error.message);
    }
  }
}

const collector = system.spawn(Props.create(() => new ReplyCollector()), 'collector');

// Make a unary call — the reply is delivered to `target`.
client.tell({ kind: 'unary', method: 'GetSensor', request: { id: 'rt-7' }, target: collector });
```

Der Endpoint ist ein einzelner `host:port`-String über
`withEndpoint`.  Jeder Call benennt einen **`target`**-Actor; der
Actor routet die Antwort und alle Stream-Frames dorthin.  Ein
Deadline pro Call ist mit `.withDeadlineMs(30000)` konfigurierbar
(der Standardwert).

### Eingehende Frames

Antworten und Stream-Frames treffen beim `target`-Actor als
`GrpcInbound`-Discriminated-Union ein:

| `kind` | Felder | Bedeutung |
| --- | --- | --- |
| `reply` | `response` | Unärer Abschluss. |
| `stream-data` | `streamId`, `chunk` | Ein Stream-Chunk. |
| `stream-end` | `streamId` | Stream sauber geschlossen. |
| `stream-error` | `streamId`, `error` | Stream fehlgeschlagen. |
| `rpc-error` | `error` | Unärer Call / Call-Setup fehlgeschlagen. |

## Streaming-Modi

gRPC unterstützt vier Call-Typen; das Framework deckt alle ab:

| Typ | Client sendet | Server liefert |
| --- | --- | --- |
| **Unär** | Einen Request | Eine Response |
| **Server-Streaming** | Einen Request | Stream von Responses |
| **Client-Streaming** | Stream von Requests | Eine Response |
| **Bidirektionales Streaming** | Stream von Requests | Stream von Responses |

Jeder clientseitige Call ist ein Kommando, das du dem Client-Actor
per `tell` schickst.  Das `kind` wählt den Call-Typ — exakt
`unary`, `serverStream`, `bidiStart`, `bidiSend` oder `bidiClose`:

```ts
// Server-streaming: one request, N chunks routed to `target`.
client.tell({
  kind: 'serverStream',
  method: 'WatchSensor',
  request: { id: 'rt-7', limit: 5 },
  target: collector,
});

// Bi-directional: open a stream, then push chunks into it.
client.tell({ kind: 'bidiStart', method: 'Chat', target: collector });
```

`bidiStart` liefert nichts direkt zurück.  Stattdessen stellt der
Actor dem `target` einen ersten `stream-data`-Frame zu, dessen
Chunk die generierte Stream-ID als `{ __streamId }` trägt.  Lies
diese ID aus und adressiere damit anschließende Sends und den
Close:

```ts
// Inside the collector, after receiving the streamId hint:
const streamId = (message.chunk as { __streamId: number }).__streamId;

client.tell({ kind: 'bidiSend', streamId, chunk: { text: 'hello' } });
client.tell({ kind: 'bidiClose', streamId });
```

Server-Stream-Calls tragen ebenfalls eine `streamId` auf ihren
`stream-data`- / `stream-end`-Frames, sodass ein einzelner
Collector mehrere gleichzeitige Streams multiplexen kann.  Streams
bleiben offen, bis eine Seite sie schließt oder der Actor stoppt.

## TLS

TLS ist über `withCredentials` opt-in.  Wird es weggelassen,
verwenden beide Actors standardmäßig **insecure**
(`{ kind: 'insecure' }`).  Zertifikate werden als
`Uint8Array`-Werte übergeben, nicht als Dateipfade:

```ts
import { readFileSync } from 'node:fs';

// Server: supply cert + key.  Add `rootCerts` to require client
// certs (mTLS).
const grpcServerOptions = GrpcServerOptions.create()
  .withProtoPath(protoPath)
  .withPackageName('sensor.v1')
  .withServiceName('SensorService')
  .withBind('0.0.0.0:50051')
  .withHandlers({ /* … */ })
  .withCredentials({
    kind: 'tls',
    cert: readFileSync('./server.crt'),
    key: readFileSync('./server.key'),
  });
new GrpcServerActor(
  grpcServerOptions,
);

// Client: pass `rootCerts` to verify the server; add cert + key for mTLS.
const grpcClientOptions = GrpcClientOptions.create()
  .withProtoPath(protoPath)
  .withPackageName('sensor.v1')
  .withServiceName('SensorService')
  .withEndpoint('sensor.svc:50051')
  .withCredentials({
    kind: 'tls',
    rootCerts: readFileSync('./ca.crt'),
  });
new GrpcClientActor(
  grpcClientOptions,
);
```

Für Mutual TLS (mTLS) füge dem Credentials-Objekt `cert` + `key`
der Gegenseite hinzu.

## Konfiguration

Instanzbezogene Builder-Optionen haben Vorrang vor HOCON, das
wiederum Vorrang vor den eingebauten Defaults hat.  Die
HOCON-Schlüssel liegen unter `actor-ts.io.broker.grpc.server` und
`actor-ts.io.broker.grpc.client`:

```hocon
actor-ts.io.broker.grpc {
  server {
    proto-path  = "./proto/sensor.proto"
    package-name = "sensor.v1"
    service-name = "SensorService"
    bind = "0.0.0.0:50051"
  }
  client {
    proto-path  = "./proto/sensor.proto"
    package-name = "sensor.v1"
    service-name = "SensorService"
    endpoint = "sensor.svc:50051"
    deadline-ms = 30s
  }
}
```

Handler und TLS-Credentials werden über den Builder bereitgestellt,
nicht über HOCON.

## Wann gRPC

Zwei primäre Einsatzfälle:

1. **Service-zu-Service innerhalb eines Clusters**, wenn
   Protobuf-typisierte Verträge für Evolution wichtig sind.
2. **Externe Clients**, die bereits gRPC sprechen (Mobile-Apps,
   Services in anderen Sprachen).

Für interne Actor-zu-Actor-Kommunikation innerhalb eines Clusters
ist der [Cluster-Transport](/de/cluster/refs-across-nodes/)
besser — direkt über TypeScript typisiert, kein Protobuf
erforderlich.  gRPC ist für **sprachübergreifende** oder
**externe Vertrags**-Fälle.

import { Aside } from '@astrojs/starlight/components';

<Aside type="caution" title="Synchronisation der .proto-Dateien">
  ```ts
  // Client und Server laden dieselbe .proto-Datei
  ```
  Der `protoPath` muss auf beiden Seiten kompatible Protobuf-
  Schemas referenzieren.  Out-of-sync .proto-Dateien erzeugen
  verwirrende Fehler (unbekannte Field-Tags, falsche
  Message-Form).  Nimm ein gemeinsames Proto-Repo oder eine
  Codegen-Pipeline.
</Aside>

<Aside type="caution" title="Streaming hat kein Buffering">
  ```ts
  // Server-Stream pusht Chunks; wenn der Target langsam ist, stauen sie sich im Speicher
  ```
  Anders als der reguläre Outbound-Buffer des Broker-Actors
  pushen gRPC-Streams so schnell, wie das Netz erlaubt.  Für
  langsame Consumer implementiere Backpressure im Target-
  Actor.
</Aside>

<Aside type="caution" title="Langlebige Streams + Reconnect">
  Streams **überleben** einen Reconnect nicht — der Zyklus baut
  die darunterliegende Verbindung neu auf, startet aber keine
  In-Flight-Streams neu.  Setze den Call nach einem
  `BrokerConnected`-Event erneut ab.
</Aside>
