`ConsumerController` ist die Empfänger-Seite zuverlässiger
Zustellung.  Er:

- Empfängt Nachrichten von einem oder mehreren `ProducerController`n.
- Ordnet nach Sequence Number.
- Dedupliziert Duplikate (gleiche Seq).
- Liefert in Ordnung an einen **Delegate**-Actor.
- Sendet ACKs zurück an den Producer, nachdem verarbeitet wurde.

```ts
import { ConsumerController } from 'actor-ts';

class OrderProcessor extends Actor<DeliveryEnvelope<OrderEvent>> {
  override async onReceive(msg: DeliveryEnvelope<OrderEvent>): Promise<void> {
    await this.process(msg.payload);
    msg.ack();   // an den Producer bestätigen
  }
}

const processor = system.actorOf(Props.create(() => new OrderProcessor()));

const consumer = system.actorOf(
  ConsumerController.props<OrderEvent>({
    consumerId: 'order-consumer',
    delegate:   processor,
  }),
);
```

## Konfiguration

```ts
interface ConsumerControllerSettings<T> {
  consumerId:      string;
  delegate:        ActorRef<DeliveryEnvelope<T>>;
  bufferSize?:     number;                  // Default 100
}
```

| Feld | Zweck |
| --- | --- |
| `consumerId` | Stabiler Identifier — wird für Recovery gebraucht. |
| `delegate` | Der Actor, der die eigentliche Arbeit macht.  Empfängt `DeliveryEnvelope<T>`. |
| `bufferSize` | Interner Buffer für Out-of-order-Eingang. |

## DeliveryEnvelope

```ts
interface DeliveryEnvelope<T> {
  readonly payload:    T;
  readonly seq:        number;
  readonly producerId: string;
  ack(): void;
  nack(reason?: string): void;
}
```

Das `onReceive` des Delegates bekommt das Envelope.  Ruf `ack()`
nach erfolgreicher Verarbeitung; der Producer kann die gebufferte
Nachricht dann freigeben.

### ACK vs. NACK

```ts
override onReceive(msg: DeliveryEnvelope<OrderEvent>): void {
  try {
    this.process(msg.payload);
    msg.ack();
  } catch (e) {
    msg.nack(`error: ${(e as Error).message}`);
    // Producer wird retransmitten
  }
}
```

`ack` gibt die Nachricht aus dem Buffer des Producers frei.

`nack` sagt dem Producer "ich habe es nicht geschafft;
retransmitte, wenn bereit".  Der Default-ConsumerController +
ProducerController des Frameworks behandeln NACK als "warten +
retransmitten"; manche Custom-Setups handhaben es anders.

### Idempotente Verarbeitung

```ts
class IdempotentConsumer extends Actor<DeliveryEnvelope<OrderEvent>> {
  private highWatermark = 0;

  override onReceive(msg: DeliveryEnvelope<OrderEvent>): void {
    if (msg.seq <= this.highWatermark) {
      // Schon verarbeitet; bestätigen, um den Producer freizugeben
      msg.ack();
      return;
    }
    this.process(msg.payload);
    this.highWatermark = msg.seq;
    msg.ack();
  }
}
```

Dedup via `highWatermark`.  Kombiniert mit Persistenz des
Watermarks ergibt das **effectively-once**-Verarbeitung.

## Out-of-order-Behandlung

```
Producer sendet 1, 2, 3
Netzwerk liefert: 1, 3, 2
ConsumerController buffert 3 (wartet auf 2)
Wenn 2 ankommt → 2 liefern, dann 3 liefern
```

Der interne Buffer hält Out-of-order-Nachrichten, bis die Lücke
gefüllt ist.  Begrenzt durch `bufferSize`.

Wenn Lücken sich nicht füllen (Producer crashed, Nachricht
endgültig verloren):

```
Wird im Buffer gehalten, bis bufferSize überschritten ist
→ die älteste blockierende Nachricht wird (still) verworfen
→ die nächsten Nachrichten können weiterlaufen
```

Für Workloads, in denen das zählt, **vertrau auf die
Retransmit-Logik des Producers** — der Producer wird unbestätigte
Nachrichten weiterhin senden.

## Failure-Semantik

```
Consumer crasht während der Verarbeitung:
  - Nachricht wurde nicht geACKt
  - Producer retransmittiert nach Timeout
  - Neuer Consumer (oder restarteter alter) verarbeitet erneut
  - Idempotenter Handler dedupliziert via highWatermark
```

**at-least-once**-Zustellung; Idempotenz liegt in deiner
Verantwortung.

## Mehrere Producer

```ts
const consumer = system.actorOf(ConsumerController.props({
  consumerId: 'multi-input',
  delegate:   processor,
}));

// Mehrere Producer senden an denselben Consumer:
const p1 = system.actorOf(ProducerController.props({ consumer: consumerRef, producerId: 'p1' }));
const p2 = system.actorOf(ProducerController.props({ consumer: consumerRef, producerId: 'p2' }));
```

Jeder Producer pflegt seine eigene Seq.  Der Consumer
dedupliziert **pro Producer** (highWatermark pro producerId):

```ts
class MultiProducerConsumer extends Actor<DeliveryEnvelope<T>> {
  private watermarks = new Map<string, number>();

  override onReceive(msg: DeliveryEnvelope<T>): void {
    const hwm = this.watermarks.get(msg.producerId) ?? 0;
    if (msg.seq <= hwm) { msg.ack(); return; }
    this.process(msg.payload);
    this.watermarks.set(msg.producerId, msg.seq);
    msg.ack();
  }
}
```

## Cluster-aware

Der `delegate` kann ein **Remote-Actor** sein.  Der Consumer
Controller routet Envelopes über den Cluster-Transport;
Serialisierung passiert an der Cluster-Grenze.

In der Praxis: Consumer + Delegate **co-located halten** —
eine cross-cluster Delegate-Ref hat unnötigen Overhead.
