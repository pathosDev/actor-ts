`cached` umhüllt einen Handler mit **Response-Caching**, verschlüsselt
über eine selbst gelieferte `key(req)`-Funktion.  Ein Cache-Treffer
liefert die gespeicherte Response, ohne den darunterliegenden Handler
aufzurufen.

```ts
import { cached, path, get } from 'actor-ts/http';
import { InMemoryCache } from 'actor-ts';

const productCache = cached({
  cache: new InMemoryCache(),
  ttlMs: 30_000,
  key:   (req) => `products:${pathParam(req, 'id')}`,
});

const routes = path('api',
  path('products',
    get(productCache(async (req) => /* expensive lookup */)),
  ),
);
```

30 Sekunden Cache pro Produkt-ID.  Der erste Request trifft den
Handler; nachfolgende GETs in den nächsten 30 s liefern die gecachte
Response.

## Konfiguration

```ts
interface ResponseCacheOptions {
  cache:          Cache;
  ttlMs:          number;
  key:            (req: HttpRequest) => string | Promise<string>;
  keyPrefix?:     string;                            // default 'rsp:'
  cacheStatuses?: ReadonlyArray<number>;             // default [200]
}
```

| Feld | Zweck |
| --- | --- |
| `cache` | Das [Cache](/de/cache/overview/)-Backend.  `InMemoryCache` für Single-Node; Redis für geteilt. |
| `ttlMs` | Wie lange jede gecachte Response gültig bleibt. |
| `key(req)` | Leitet den Cache-Key aus dem Request ab — enthält typischerweise URL-Params, Tenant-ID, Accept-Header usw.  **Pflicht.** |
| `keyPrefix` | Cache-Key-Namespace.  Default `'rsp:'`, damit sich mehrere Response-Caches im selben Redis nicht überschneiden. |
| `cacheStatuses` | Welche Status-Codes cachebar sind.  Default `[200]` — nur 2xx-Responses werden gecacht. |

## Cache-Key-Zusammensetzung

Die `key(req)`-Funktion ist der gesamte Cache-Key-Vertrag — was du
dort aufnimmst, bestimmt, was separat gecacht wird.  Gängige Muster:

```ts
// Per-URL-path key:
key: (req) => req.url.pathname,

// Per-tenant:
key: (req) => `${req.headers['x-tenant-id'] ?? 'public'}:${req.url.pathname}`,

// Per-locale + content type:
key: (req) => `${req.headers['accept-language']}:${req.headers['accept']}:${req.url.pathname}`,
```

Baue den Key so, dass er über äquivalente Requests deterministisch,
aber über getrennt zu cachende Responses eindeutig ist.

## Wann cachen

```
GET /api/products       → cache (stable, read-heavy)
POST /api/orders        → don't wrap with cached() (mutation)
GET /api/users/me       → cache only with per-user key
GET /api/feed?cursor=X  → cache (one entry per cursor)
```

Der Wrapper ist eine Handler-Ebenen-Entscheidung — du wählst, welche
Handler du umhüllst.  Mutationen sollten typischerweise
[`idempotent`](/de/http/middleware/idempotency-key/) verwenden.

## Cache-Invalidierung

`cached` **invalidiert nicht automatisch** bei Writes — die
Veraltung ist durch `ttlMs` begrenzt.

Für **stärkere Konsistenz** zwei Muster:

### Kurze TTLs

```ts
cached({ cache, ttlMs: 1_000, key });   // 1 second
```

Sub-Sekunden-TTLs begrenzen die Veraltung.  Gut für „echtzeit-nahe"
APIs.

### Manuelle Invalidierung

```ts
// After a write that affects products:42, invalidate the cache:
await cache.delete('rsp:products:42');   // 'rsp:' is the default keyPrefix
```

Der Cache ist einfach ein `Cache` — du kannst den mit Prefix
versehenen Key direkt `delete`n, wenn ein Write ihn invalidieren soll.

## Geteilter Cache über Pods

```ts
import { RedisCache, RedisCacheOptions } from 'actor-ts';

const redisCacheOptions = RedisCacheOptions.create().withUrl('redis://...');
const productCache = cached({
  cache: new RedisCache(redisCacheOptions),
  ttlMs: 30_000,
  key:   (req) => `products:${pathParam(req, 'id')}`,
});
```

Mit einem Redis-gestützten Cache teilen sich alle Pods denselben
Cache — nützlich für Multi-Pod-Deployments, in denen du konsistentes
Caching willst.

Für **Per-Pod-Caches** (jeder Pod hat seinen eigenen InMemoryCache)
können Pods kurzzeitig unterschiedliche gecachte Responses liefern,
bis sich die TTLs angleichen.  Oft akzeptabel; hängt von den
Konsistenzanforderungen ab.

## Verschiedene Status-Codes cachen

```ts
cached({
  cache, ttlMs: 30_000, key,
  cacheStatuses: [200, 404],   // cache both hits AND "not found"s
});
```

Cache `404`s, wenn du viele Lookups für nicht existierende IDs hast —
spart wiederholte DB-Zugriffe.  Default ist nur `[200]`, damit ein
transientes 500 nicht 30 Sekunden lang hängen bleibt.

import { Aside } from '@astrojs/starlight/components';

<Aside type="caution" title="Authentifizierte Responses cachen">
  ```ts
  // GET /api/users/me with cache → all users see the first user's response
  ```
  Ohne den User in `key(req)` aufzunehmen, lecken gecachte Responses
  über User hinweg.  Nimm entweder `hashUser(req.headers.authorization)`
  in den Key auf oder umhülle Per-User-Endpunkte gar nicht mit
  `cached`.
</Aside>

<Aside type="note" title="Stampede-Schutz ist Per-Process">
  ```ts
  // TTL expires; 1000 concurrent requests on one pod → the handler runs once
  ```
  Bei einem Cache-Miss führt der Wrapper den Handler **einmal pro
  Prozess** aus, und die übrigen gleichzeitigen Requests für denselben
  Key warten auf dieses eine Ergebnis (eine In-Process-Single-Flight-
  Map).  Es ist **nicht** cluster-weit: mit N Pods kann der Handler bis
  zu N-mal gleichzeitig laufen.  Stale-While-Revalidate ist nicht
  eingebaut — implementiere es bei Bedarf selbst.
</Aside>

<Aside type="caution" title="In-Memory-Cache auf dein Response-Set dimensionieren">
  ```ts
  new InMemoryCache();                        // LRU-bounded: 10 000 entries
  new InMemoryCache({ maxEntries: 100_000 }); // raise it for a larger response set
  ```
  `InMemoryCache` ist standardmäßig LRU-begrenzt (10 000 Einträge), ein
  Response-Cache kann also nicht unbegrenzt wachsen — aber eine zu
  kleine Schranke evictet heiße Responses zu früh.  Dimensioniere
  `maxEntries` auf die Zahl der erwarteten eindeutigen Cache-Keys.
  Siehe [In-Memory-Cache](/de/cache/in-memory/).
</Aside>
