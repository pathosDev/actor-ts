The framework's [envelope format](/persistence/migration/envelope-format/)
only kicks in when an adapter is configured.  **Before** you set an
adapter, events are stored **raw**:

```json
{ "kind": "deposited", "amount": 100 }
```

Once an adapter is in place, new events are wrapped in an envelope:

```json
{ "_v": 1, "_t": "BankAccount.deposited", "_e": { "kind": "deposited", "amount": 100 } }
```

That creates a problem when you add an adapter to an **existing
journal**: the old raw events have no envelope, but the adapter
expects one.  Reading the first raw event throws `MigrationError`.

The fix is a **one-shot bulk migration**: walk the journal once,
wrap every raw event as a version-1 envelope, write it back.  From
then on your normal
[defaultsAdapter](/persistence/migration/default-adapter/) /
[migratingAdapter](/persistence/migration/migrating-adapter/)
replays them like any other v1 event.  This is a real rewrite of
the stored data, not a read-time shim.

## In-memory journal

For the in-memory journal, `migrateInMemoryJournal` does the whole
walk-and-rewrite.  You supply a `manifestFor` that derives the
stable `_t` discriminator from each event:

```ts
import { migrateInMemoryJournal, formatMigrationResult } from 'actor-ts';

const result = await migrateInMemoryJournal(
  journal,
  (e: { kind: string }) => `BankAccount.${e.kind}`,
);

console.log(formatMigrationResult('events', result));
// â†’ "events: 3 wrapped, 0 already enveloped, 3 inspected"
```

Sequence numbers, timestamps, and tags are preserved â€” only the
event payload is wrapped.  Run this **before** the actor with the
new adapter recovers.

## Other backends

`migrateInMemoryJournal` relies on an internal `_remapForMigration`
hook that only the in-memory journal exposes.  Cassandra / SQLite /
Postgres / S3 journals each need a backend-specific rewrite path
(SQL `UPDATE`, CQL `UPDATE`, S3 `PUT`).  Use the pure per-row
primitive `wrapEventAsEnvelope` as the building block:

```ts
import { wrapEventAsEnvelope } from 'actor-ts';

// Inside your own per-row rewrite loop:
for (const row of rows) {
  const enveloped = wrapEventAsEnvelope(
    row.event,
    (e) => `BankAccount.${e.kind}`,
  );
  await backend.rewrite(row.id, enveloped);
  // enveloped === { _v: 1, _t: 'BankAccount.deposited', _e: row.event }
}
```

`wrapEventAsEnvelope(event, manifestFor, version?)` is pure and
**idempotent** â€” an event that already looks like an envelope is
returned unchanged, so re-running the migration is safe.

## Snapshots

Legacy raw snapshots have the same problem.  `migrateSnapshotStore`
wraps the latest snapshot per persistence id (sourced from the
journal), using `wrapStateAsEnvelope` under the hood:

```ts
import { migrateSnapshotStore } from 'actor-ts';

const pids   = await journal.persistenceIds();
const result = await migrateSnapshotStore(store, pids, (s) => 'BankAccount.State');
```

## After the migration

Once the journal is migrated, the raw events are v1 envelopes.
The actor uses a **normal adapter** â€” there's no special
"legacy-wrapping" adapter at read time:

```ts
class Account extends PersistentActor<Command, EventV2, State> {
  override eventAdapter(): EventAdapter<EventV2> {
    return defaultsAdapter<EventV2>({
      manifest:       'BankAccount.deposited',
      currentVersion: 2,
      defaults:       { 1: { currency: 'USD' } },
    });
  }
}
```

The `manifest` here must match the `_t` your `manifestFor` produced
during the migration.

## Signatures

```ts
function wrapEventAsEnvelope<E>(
  event: E, manifestFor: (e: E) => string, version?: number,   // version default 1
): JournalEnvelope<E>;

function wrapStateAsEnvelope<S>(
  state: S, manifestFor: (s: S) => string, version?: number,
): JournalEnvelope<S>;

function migrateInMemoryJournal<E>(
  journal: Journal, manifestFor: (e: E) => string, opts?: { version?: number },
): Promise<MigrationResult>;

function migrateSnapshotStore<S>(
  store: SnapshotStore, pids: ReadonlyArray<string>,
  manifestFor: (s: S) => string, opts?: { version?: number },
): Promise<MigrationResult>;

interface MigrationResult {
  inspected: number;   // entries examined
  wrapped:   number;   // raw â†’ envelope
  skipped:   number;   // already enveloped, left untouched
}
```

## Pitfalls

import { Aside } from '@astrojs/starlight/components';

<Aside type="caution" title="Run the migration before recovery">
  ```ts
  // Deploy the adapter without migrating first â†’ first raw event throws
  override eventAdapter() { return defaultsAdapter({ ... }); }
  ```
  The bulk migration must run against the journal **before** an
  actor with the new adapter recovers â€” otherwise the first raw
  event throws `MigrationError`.
</Aside>

<Aside type="caution" title="`manifestFor` must be stable and match the adapter">
  ```ts
  // migration wrote _t: 'BankAccount.deposited'
  // but the adapter's manifest is 'Deposited' â†’ MigrationError on read
  ```
  The `_t` your `manifestFor` produces is baked into the stored
  envelope.  It must equal the `manifest` your adapter declares, and
  it must be derived deterministically from the event.
</Aside>

<Aside type="caution" title="Custom backends need their own rewrite">
  ```ts
  await migrateInMemoryJournal(cassandraJournal, manifestFor);
  // âś— throws: journal does not expose _remapForMigration
  ```
  Only the in-memory journal supports `migrateInMemoryJournal`.  For
  other backends, write a few lines of glue around
  `wrapEventAsEnvelope` and your store's native update path.
</Aside>
