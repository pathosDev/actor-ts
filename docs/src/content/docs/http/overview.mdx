title: HTTP overview
description: A typed route DSL with three pluggable backends (Fastify, Bun.serve, Express), a shared HTTP client, and middleware for caching, rate-limiting, and idempotency. Built as an extension; not on top of `BrokerActor`.
---

The HTTP module is **separate from the I/O broker actors** ‚Äî HTTP
servers don't fit the "one connection many messages" shape, so
they get their own DSL and backend abstraction.

```ts
import { ActorSystem } from 'actor-ts';
import { path, get, post, completeJson, entity, concat } from 'actor-ts/http';

const system = ActorSystem.create('my-app');

const routes = path('api',
  path('orders',
    concat(
      get(async () => completeJson(200, { orders: [] })),
      post(async (req) => {
        const order = entity<NewOrder>(req);
        // ... handle ...
        return completeJson(201, { id: 'o-1' });
      }),
    ),
  ),
);

const binding = await system.http(8080).bind(routes);
console.log(`bound on ${binding.host}:${binding.port}`);
```

Three things going on:

- **The route DSL** ‚Äî `path`, `get`, `post`, `concat` compose a
  tree of routes.  Type-safe; compiles to a flat list at bind time.
- **The marshaller** ‚Äî `entity<T>(req)` decodes the request body
  by Content-Type; `completeJson` / `completeText` encode the
  response.
- **The shortcut** ‚Äî `system.http(port).bind(routes)` starts a
  server with the default Fastify backend.  For a different
  backend, pass `backend:` (see below).

The DSL builds an in-memory route tree; `system.http(...).bind(routes)`
flattens it and registers everything with the chosen backend.

## The backends

```ts
import { HonoBackend } from 'actor-ts/http';

// Default ‚Äî no setup needed; Fastify is a hard dependency:
await system.http(8080).bind(routes);

// Explicit override (Express / Hono are peer-deps; opt-in):
await system.http(8080, { backend: new HonoBackend() }).bind(routes);

// For full control, the underlying extension surface is still there:
import { HttpExtensionId } from 'actor-ts';
await system.extension(HttpExtensionId)
  .newServerAt('0.0.0.0', 8080)
  .useBackend(new HonoBackend())
  .bind(routes);
```

| Backend | Runtime fit | When |
| --- | --- | --- |
| **`FastifyBackend`** *(default)* | Bun, Node | Production default ‚Äî Fastify is the most-battle-tested. |
| **`ExpressBackend`** | Node, Bun | Existing Express middleware ecosystem fits cleanly. |
| **`HonoBackend`** | Bun, Node, Deno | Tiny + edge-friendly; the most Bun-and-Deno-native option. |

All three implement the same `HttpServerBackend` interface ‚Äî the
route DSL compiles identically; the backend just owns the
listen/dispatch loop.

See the [backend pages](/http/backends/fastify/) for the
configuration options each accepts.

## The HTTP client

```ts
const response = await http.singleRequest({
  method: 'POST',
  url:    'https://api.example.com/orders',
  headers: { 'content-type': 'application/json' },
  body:   JSON.stringify({ sku: 'book-1' }),
});

console.log(response.status, response.body);
```

The shared client wraps the runtime's native `fetch`.  Works
identically on Bun, Node, and Deno (every supported runtime
has fetch built in).

For more elaborate client patterns (retries, caching), wrap the
client in a [retry](/patterns/retry/) +
[circuit-breaker](/patterns/circuit-breaker/) call.

## Middleware

```ts
import { cached, rateLimit } from 'actor-ts/http';

const routes = concat(
  rateLimit({ max: 100, intervalMs: 1_000 })(
    cached({ ttlMs: 30_000 })(
      path('api', /* ... */),
    ),
  ),
);
```

The framework ships three:

| Middleware | What it does |
| --- | --- |
| **Response cache** | Caches GET responses keyed by URL + Vary headers. |
| **Rate limit** | Per-IP / per-key token-bucket rate limiter. |
| **Idempotency key** | De-duplicates writes based on an `Idempotency-Key` header. |

Each is a `Route -> Route` transformer ‚Äî wraps a sub-tree of routes
with its behavior.  Compose by nesting.  See
[middleware pages](/http/middleware/response-cache/).

## Connecting HTTP to actors

The route handler returns a `Promise<HttpResponse>` ‚Äî you can
freely `await` actor calls inside:

```ts
import {} from 'actor-ts';

const routes = path('orders',
  path(':id',
    get(async (req) => {
      const id = req.path.split('/').pop();
      const order = await orderRegistry.ask({ kind: 'get', id },
        5_000,);
      return completeJson(200, order);
    }),
  ),
);
```

This is **the** common pattern ‚Äî HTTP handlers act as a thin
adapter: parse the request, ask an actor, marshal the reply.  Keep
business logic in actors; HTTP handlers stay short.

import { Aside } from '@astrojs/starlight/components';

<Aside type="caution" title="Don't share state in route handlers">
  ```ts
  let cache: Record<string, Order> = {};   // ‚úó shared mutable state
  ```
  Route handlers can run concurrently ‚Äî multiple requests in
  flight.  Plain mutable state outside actors causes the same
  races as anywhere else in JavaScript.  If you need
  concurrent-safe state, put it in an actor and ask it.
</Aside>

<Aside type="caution" title="Slow handlers block the HTTP server">
  ```ts
  get(async () => {
    await heavyComputation();   // 2 seconds
    return completeJson(200, ...);
  });
  ```
  All three backends serve handlers concurrently, but a slow
  handler still occupies a connection's lifetime ‚Äî pile up
  enough of them and the server runs out of capacity.  Offload
  heavy work to an actor and use `ask` with a sane timeout, or
  use a job-queue pattern with a job-status endpoint.
</Aside>

<Aside type="caution" title="Backend-specific quirks">
  Each backend has its own pre-existing ecosystem: Fastify's
  schema validation, Bun.serve's WebSocket integration, Express
  middleware.  These are **not** exposed through the DSL ‚Äî you
  reach for them by configuring the backend instance directly
  before passing to `useBackend`.  Cross-backend portability
  drops as soon as you do this.
</Aside>
