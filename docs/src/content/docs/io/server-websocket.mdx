---

Server-side WebSocket is **part of the HTTP route DSL**.  The
`websocket()` directive upgrades matching requests and wires every
connection to a single `WebSocketServerActor` — the hub — which
you implement.  For **outbound** client connections, see
[WebSocketClientActor](/io/websocket/).

```ts
import {
  ActorSystem, Props, HttpExtensionId,
  WebSocketServerActor, websocket, type WsConnection,
} from 'actor-ts';
import { match } from 'ts-pattern';

type ClientMsg = { kind: 'setName'; name: string } | { kind: 'say'; text: string };
type ServerMsg = { kind: 'system'; text: string } | { kind: 'chat'; from: string; text: string };

class ChatRoom extends WebSocketServerActor<ServerMsg, ClientMsg> {
  private readonly names = new Map<string, string>();

  onMessage(msg: ClientMsg): void {
    match(msg)
      .with({ kind: 'setName' }, ({ name }) => {
        this.names.set(this.connection.id, name);
        this.reply({ kind: 'system', text: `hi ${name}` });
      })
      .with({ kind: 'say' }, ({ text }) => {
        this.broadcast({ kind: 'chat', from: this.names.get(this.connection.id) ?? 'anon', text });
      })
      .exhaustive();
  }

  override onClientDisconnected(c: WsConnection<ServerMsg>): void {
    this.names.delete(c.id);
  }
}

const system = ActorSystem.create('chat');
const chat = system.spawn(Props.create(() => new ChatRoom()), 'chat');

await system.extension(HttpExtensionId)
  .newServerAt('0.0.0.0', 8080)
  .bind(websocket('/ws', chat));
```

The hub's type parameters read from the **server's** point of
view: `TOut` = the messages the server sends, `TIn` = the decoded
messages it receives.

## The `websocket()` directive

`websocket()` produces a `Route`, so it composes with the rest of
the [route DSL](/http/route-dsl/) — `path()`, `concat()`, and
`withMiddleware()`.  Two forms:

```ts
import { websocket, path, concat } from 'actor-ts';

// 1. Bare directive — mount it under a path yourself:
path('ws', websocket(chat));

// 2. Path sugar — equivalent to path(p, websocket(target)):
websocket('/ws', chat);

// Mixed with normal HTTP routes on the same server:
concat(
  path('api', apiRoutes),
  websocket('/ws', chat),
);
```

### Options

```ts
interface WebSocketRouteOptions<TOut, TIn> {
  codec?:            WsCodec<TOut, TIn>;             // default jsonCodec()
  maxFrameBytes?:    number;                          // default 1 MiB
  onOversizeFrame?:  'close' | 'drop';                // default 'close' (1009)
  onInvalidMessage?: 'close' | 'drop' | 'hook';       // default 'close' (1003)
  maxBufferedBytes?: number;                          // default 4 MiB
  onBackpressure?:   'drop' | 'close';                // default 'drop'
}
```

```ts
const webSocketRouteOptions = WebSocketRouteOptions.create()
  .withMaxFrameBytes(256 * 1024)
  .withOnOversizeFrame('close')       // close with 1009 Message Too Big
  .withOnInvalidMessage('close');
websocket('/ws', chat, webSocketRouteOptions);    // close with 1003 Unsupported Data
```

The frame-size cap is enforced on the **raw frame before decode**.
`onInvalidMessage: 'hook'` routes decode failures to the actor's
`onInvalidMessage` hook instead of closing.

### Middleware runs at upgrade time

`withMiddleware()` composes with `websocket()`, but the middleware
runs **once, at upgrade time, against the HTTP upgrade request** —
not per WebSocket message.  A rejecting middleware returns a normal
HTTP error and the upgrade never completes, so auth middleware
like `BearerTokenAuth` or `IpAllowlist` gates the handshake:

```ts
import { websocket, withMiddleware } from 'actor-ts';

withMiddleware(BearerTokenAuth({ /* ... */ }), websocket('/ws', chat));
// A bad token → HTTP 401, no upgrade.  A good token → the socket opens.
```

See the [HTTP overview](/http/overview/) for the middleware set.

## `WebSocketServerActor`

One actor per route — the **hub**.  It sees every connection's
events, serialized:

```ts
abstract class WebSocketServerActor<TOut, TIn, TSelf = never> {
  // You implement:
  abstract onMessage(msg: TIn): void | Promise<void>;

  // Optional overrides:
  protected onClientConnected(client: WsConnection<TOut>): void;
  protected onClientDisconnected(client: WsConnection<TOut>, info: WsCloseInfo): void;
  protected onInvalidMessage(client: WsConnection<TOut>, error: WsDecodeError): void;
  protected onSelfMessage(msg: TSelf): void;
}
```

### Helpers

Inside `onMessage` and the hooks, these are available:

| Member | What it does |
| --- | --- |
| `this.connection` | The `WsConnection<TOut>` whose event is being processed. |
| `this.reply(msg)` | Send `msg` to the current connection. |
| `this.broadcast(msg, filter?)` | Send to every connection (optionally filtered by predicate). |
| `this.clients` | `ReadonlyMap<string, WsConnection<TOut>>` of all live connections. |
| `this.closeAll(code?, reason?)` | Close every connection. |

```ts
onMessage(msg: ClientMsg): void {
  this.reply({ kind: 'system', text: 'got it' });         // → the sender
  this.broadcast({ kind: 'chat', from: 'x', text: 'hi' }); // → everyone
  this.broadcast(notice, (c) => c.id !== this.connection.id); // → everyone else
}
```

### Per-connection ordering

Every event for a given connection is serialized through the one
hub actor in this order:

```
onClientConnected → onMessage* (in frame order) → onClientDisconnected
```

`onClientConnected` runs once, then zero or more `onMessage`
calls in frame order, then exactly one `onClientDisconnected`.
Because it all runs on a single actor, you get the actor model's
usual guarantee — no concurrent handler execution, no locks.

<Aside type="caution" title="`onReceive` is sealed">
  Don't override `onReceive` on a `WebSocketServerActor` — it is
  sealed.  For every connection the framework spawns an **internal
  connection actor as a child of the server actor** (the tree is
  `server → conn-1, conn-2, …`); that child owns the socket, decodes
  inbound frames, writes outbound, and is stopped automatically when
  the socket closes.  You never see or manage it — implement
  `onMessage` and the hooks instead.
</Aside>

## `WsConnection`

Each connection is a `WsConnection<TOut>`, which extends
`ActorRef<TOut>`:

```ts
interface WsConnection<TOut> extends ActorRef<TOut> {
  readonly id: string;
  readonly remoteAddress?: string;
  readonly upgrade: WsUpgradeInfo;
  readonly isOpen: boolean;
  tell(msg: TOut): void;         // encode via codec + send
  sendRaw(frame: WsFrame): void; // bypass the codec
  close(code?: number, reason?: string): void;
}
```

`upgrade` carries the handshake context:

```ts
type WsUpgradeInfo = {
  path: string;
  params: Record<string, string>;
  query: Record<string, string>;
  headers: Record<string, string>;
  remoteAddress?: string;
  subprotocol?: string;
};
```

`WsCloseInfo` (passed to `onClientDisconnected`) is
`{ code: number; reason: string; initiatedBy: 'client' | 'server' | 'error' }`.

<Aside type="caution" title="No deathwatch on connections">
  `WsConnection` is a ref, but you **cannot** deathwatch it.  To
  learn when a client goes away, override `onClientDisconnected`
  — it's the single, ordered end-of-life signal per connection.
</Aside>

## Codecs

The route codec decodes inbound frames into `TIn` and encodes
`TOut` replies.  Default is `jsonCodec()`:

```ts
import { jsonCodec, WebSocketRouteOptions } from 'actor-ts';

const webSocketRouteOptions = WebSocketRouteOptions.create().withCodec(jsonCodec<ServerMsg, ClientMsg>({
    validate: (v: unknown): ClientMsg => ClientMsgSchema.parse(v),
  }));
websocket('/ws', chat, webSocketRouteOptions);
```

For binary protocols, `rawCodec()` gives you raw frames
(`TOut = TIn = WsFrame`):

```ts
import { rawCodec, WebSocketRouteOptions, WebSocketServerActor, type WsFrame } from 'actor-ts';

const webSocketRouteOptions = WebSocketRouteOptions.create().withCodec(rawCodec());
class BinaryHub extends WebSocketServerActor<WsFrame, WsFrame> {
  onMessage(frame: WsFrame): void {
    if (frame.kind === 'binary') this.reply({ kind: 'binary', data: process(frame.data) });
  }
}

websocket('/stream', binaryHub, webSocketRouteOptions);
```

Decode failures throw `WsDecodeError` and follow the route's
`onInvalidMessage` policy.

## Backends and runtimes

`websocket()` works on all three HTTP backends:

| Backend | Peer dependency | Runtime notes |
| --- | --- | --- |
| Fastify (default) | `@fastify/websocket` | Runs on Bun and Node; on Deno prefer Hono. |
| Express | `ws` | — |
| Hono | Bun & Deno: built into `hono`.  Hono-on-Node: `@hono/node-ws`. | Per-runtime helpers. |

```bash
npm install @fastify/websocket   # Fastify (default)
npm install ws                   # Express
npm install @hono/node-ws        # Hono on Node only
```

## HOCON config

Route defaults live under `actor-ts.http.websocket`:

```hocon
actor-ts.http.websocket {
  maxFrameBytes    = 1048576
  onOversizeFrame  = close
  onInvalidMessage = close
  maxBufferedBytes = 4194304
  onBackpressure   = drop
}
```

Precedence: **route options > HOCON > built-in defaults**.

## When to use it

Three good fits:

1. **Real-time UIs** — pushing updates to browser clients.
2. **Custom messaging protocols** — game servers, chat backends.
3. **Binary streaming** — use `rawCodec()` and handle frames
   directly.

For one-way streams from server to client, [SSE](/io/sse/) is
simpler.  For request/reply RPC, plain [HTTP](/http/overview/) is
the norm.

import { Aside } from '@astrojs/starlight/components';

<Aside type="note" title="Renamed from ServerWebSocketActor">
  This replaces the old `ServerWebSocketActor` and its
  `serverWebSocketActorOf` / `bunWebSocketHandlers` helpers.
  Connections are no longer per-connection *child* actors you
  spawn — the hub sees every connection through one actor, and the
  framework manages an internal session actor per socket.
</Aside>

<Aside type="caution" title="Backpressure on slow clients">
  ```ts
  const routeOptions = WebSocketRouteOptions.create()
    .withMaxBufferedBytes(4 * 1024 * 1024)
    .withOnBackpressure('drop');
  websocket('/ws', chat, routeOptions);
  ```
  When a client can't keep up, its send buffer grows.  Past
  `maxBufferedBytes` the connection either drops frames
  (`onBackpressure: 'drop'`, default) or is closed
  (`'close'`).  Broadcasts to many slow clients are where this
  bites — pick the policy that fits your protocol.
</Aside>
